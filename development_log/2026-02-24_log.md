## 一、手机端水合错误修复 | Mobile Hydration Fix

- **现象**：在手机端打开页面时出现 Next.js 水合错误，提示「A tree hydrated but some attributes of the server rendered HTML didn't match the client properties」，常见原因包括服务端/客户端分支（如 `typeof window`）、`Date.now()`/`Math.random()`、或**用户区域/时区下的日期格式**与服务器不一致。
- **原因**：`DataAnalysisSection.tsx` 在组件顶层用 `new Date()` 和 `toLocaleDateString('en-US', ...)` 生成月份/季度/年份选项（`buildMonthOptions` / `buildQuarterOptions` / `buildYearOptions`）。服务端与手机端时区不同时，生成的选项列表不一致，导致首屏 DOM 不匹配。
- **修改**：将 `monthOptions`、`quarterOptions`、`yearOptions` 改为 `useState` 初始空数组，在 **`useEffect` 中（仅客户端执行）** 再调用上述 build 函数并 `setState`。这样服务端与首屏客户端渲染一致（无时间相关选项），挂载后再填充，避免水合不一致。
- **文件**：`frontend/app/dashboard/DataAnalysisSection.tsx`

## 二、「Load failed」错误提示优化 | Load Failed Error Message

- **现象**：登录成功后，在手机通过 ngrok 访问前端时出现「Load failed」或 Console TypeError，上传/相机上传均失败。
- **原因**：前端 `NEXT_PUBLIC_API_URL` 默认或仍为 `http://localhost:8000`。手机端「localhost」指向手机本机，无法访问开发机上的后端，fetch 失败；iOS/Safari 常报「Load failed」。
- **修改**：
  - **dashboard 上传**（`frontend/app/dashboard/page.tsx`）：在 catch 中识别 `error.message` 为 `"Load failed"`、`"Load failed."` 或包含 `"Failed to fetch"` 时，不再只显示原文，而是显示说明：若用手机通过 ngrok 访问，需把**后端也用 ngrok 暴露**，并在前端 `.env.local` 中设置 `NEXT_PUBLIC_API_URL` 为**后端的 ngrok 地址**，并附带当前 API 地址便于排查。
  - **相机上传**（`frontend/app/dashboard/camera/CameraCaptureButton.tsx`）：新增 `normalizeNetworkError()`，对「Load failed」等做同样友好提示。
- **文件**：`frontend/app/dashboard/page.tsx`，`frontend/app/dashboard/camera/CameraCaptureButton.tsx`

## 三、相机流程：拍照 → 预览 → 确认上传 | Camera: Preview Before Upload

- **原行为**：点击「Take photo」后直接上传，无确认步骤；请求失败时界面反馈弱，用户感觉「毫无反应」。
- **新行为**：
  1. 点击「Take photo」→ 截取当前帧，画面**冻结**为一张预览图。
  2. 预览下方三个按钮：**Upload**（确认上传）、**Retake**（重拍）、**Cancel**（关闭不上传）。
  3. 仅当用户点击 **Upload** 时才发起 `/api/receipt/workflow` 请求；失败时走上述「Load failed」友好提示。
- **实现要点**：
  - 新增状态：`capturedBlob`、`previewUrl`（`URL.createObjectURL(blob)` 用于展示，关闭或 Retake 时 `URL.revokeObjectURL` 避免内存泄漏）。
  - `handleCapture` 只负责截帧、生成 blob、设置预览；`handleUpload` 负责用 `capturedBlob` 上传；`handleRetake` 清空预览回到实时画面。
- **文件**：`frontend/app/dashboard/camera/CameraCaptureButton.tsx`

## 四、讨论记录（无代码变更）| Discussion Notes

- **ngrok 报错「The endpoint ... is already online」**：不是免费计划禁止多隧道，而是**同一端点已被占用**。解决：先关掉已占用的 ngrok 进程，再开新隧道；若需同时暴露前后端，可查 ngrok 单进程多隧道配置或付费多隧道。
- **后端部署 GCP vs 买 ngrok 会员**：后端上 GCP、前端只改 API 地址可行；但每次改后端都要部署，反馈慢。若近期修 bug 频繁，更推荐花少量费用用 ngrok 多隧道，本机改完即测。

## 五、Firebase 认证迁移与重构 | Firebase Auth Migration

认证从 **Supabase Auth** 迁到 **Firebase Auth（Email Link）**，后端优先校验 Firebase ID Token，再按 `firebase_uid` 查/建用户，实现「真人 + 同邮箱即同账号」。

### 5.1 数据库

- **042_firebase_uid_and_drop_auth_fk.sql**
  - `users` 表新增 `firebase_uid`（唯一、可空），用于存 Firebase UID（ID token 的 `sub`）。
  - 去掉 `users.id` 对 `auth.users` 的外键，以便仅用 Firebase 登录的用户也能建 `users` 行（id 用 `gen_random_uuid()`）。
- **一次性脚本**：`backend/database/scripts/one_time_link_firebase_to_existing_user.sql`  
  用于把「同邮箱、先 Supabase 后 Firebase」的同一人绑到一条 `users` 记录（按邮箱匹配并回填 `firebase_uid`）。

### 5.2 后端

- **Firebase 校验与用户解析**（`backend/app/services/auth/firebase_auth.py`）
  - `verify_firebase_token(id_token)`：用 `firebase_admin.auth().verify_id_token` 校验，取 `uid`/`user_id`/`sub` 与 `email`（兼容 dict/DecodedToken 两种返回）。
  - `get_or_create_user_id(firebase_uid, email)`：先按 `firebase_uid` 查用户；若无则按邮箱（精确 + 不区分大小写）匹配并回写 `firebase_uid`；再若无则新建用户，`user_class="free"`。
  - 依赖 `FIREBASE_SERVICE_ACCOUNT_PATH`（或 `GOOGLE_APPLICATION_CREDENTIALS`）指向服务账号 JSON；需安装 `firebase-admin`。
- **鉴权入口**（`backend/app/services/auth/jwt_auth.py`）
  - `get_current_user`：先尝试 Firebase（`verify_firebase_token` → `get_or_create_user_id`），成功则返回 `user_id`；失败再走 Supabase JWT（RS256/HS256）。
  - 增加调试与 fallback 日志（如 `[Auth] Trying Firebase verification`、`Firebase verify returned None`、ImportError 打 ERROR 等），便于排查 401。
- **401 排查与修复**
  - 曾出现 401：后端运行环境未安装 `firebase-admin`，Firebase 分支未执行，请求落到 Supabase JWT 校验（Google key id 不匹配）导致 401。解决：在对应 Python 环境中 `pip install firebase-admin` 并重启后端。
  - 修正 `supabase_client` 中 `payment_types` 导入路径：`..processors` → `...processors`（processors 在 `app` 下，不在 `app.services` 下）。

### 5.3 前端

- **登录**：改为 Firebase Email Link（`sendSignInLinkToEmail`），回调页用 `signInWithEmailLink` 完成登录。
- **Dashboard / Admin**：用 `onAuthStateChanged` + `getIdToken()` 获取 ID Token，请求 API 时带 `Authorization: Bearer <token>`；移除登录页 ngrok 弹窗、删除 dashboard 中残留的 `createClient()` 等 Supabase 客户端用法。
- **环境**：前端 `.env.local` 配置 Firebase 相关变量；后端 `.env` 中 `FIREBASE_SERVICE_ACCOUNT_PATH` 指向 `backend/keys/` 下服务账号 JSON。
