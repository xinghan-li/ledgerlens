## 一、ngrok 与 CORS | Ngrok & CORS

- **后端**：`config.py` 新增 `cors_origins_extra`（环境变量 `CORS_ORIGINS`），逗号分隔的额外允许 origin，用于手机通过 ngrok 访问前端时后端接受跨域请求（例如 `https://xxx.ngrok-free.app` 或 `http://192.168.x.x:3000`）。
- **main.py**：CORS 中间件在原有 allow 列表基础上，若设置了 `cors_origins_extra`，则解析并加入 `allow_origins`。
- **前后端 .env**：前端 `NEXT_PUBLIC_API_URL` 指向后端 ngrok 地址；后端 `.env` 中可配置 `CORS_ORIGINS` 为前端 ngrok 或局域网地址，便于手机真机测相机与上传。

## 三、新架构与 Escalation 流程 | Architecture & Escalation Flow

### 3.1 流程图与文档（RECEIPT_WORKFLOW_CASCADE.md）

- **Mermaid 流程图**：原「Vision/首轮失败 → Textract+OpenAI → Sum Check → needs_review」改为统一 **Escalation** 节点：
  - 当「足够多的否且 Sum Check 仍不过」时，不再走 Amazon Textract + OpenAI，而是**原图直接送入** `OPENAI_ESCALATION_MODEL`（如 gpt-5.1）与 `GEMINI_ESCALATION_MODEL`（如 gemini-3）。
  - 两模型用同一套 bookkeeper 提示词输出结构化 JSON 并自检 sum check；结果做**双模型共识**（item 数量一致、sum check 均过、total 一致则采纳，否则 escalate 给客户）。
- **说明**：若未配置上述两个环境变量，则回退到原 Textract + OpenAI 流程。§1.1 审阅中「Valid 否 / 用户确认后」分支描述同步为 Escalation（未配置则回退 Textract+OpenAI）。

### 3.2 配置（config.py）

- **OPENAI_ESCALATION_MODEL**（可选）：cascade 失败时升级到该模型，带图输入（替代 Textract+OpenAI）。
- **GEMINI_ESCALATION_MODEL**（可选）：同上，与 OpenAI escalation 双模型共识。

### 3.3 后端实现（workflow_processor / sum_checker / supabase 等）

- **workflow_processor**：接入 Escalation 分支（原图 → 双 escalation 模型）、共识逻辑与回退到 Textract+OpenAI 的兼容。
- **sum_checker**：与 escalation 结果校验、sum check 通过判定配合。
- **supabase_client**：escalation 相关状态或 run 记录（若有）。
- **prompts**：escalation 用 bookkeeper 提示词或复用现有 receipt 结构化 prompt。
- 其他：address_matcher、payment_types、store_config_loader、documentai_client、auth、receipt_categorizer、llm_client、gemini_client 等在该提交中有配合修改或扩展。
