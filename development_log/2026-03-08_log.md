## 二、新用户 401 / “Session updated. Please try again.” | New User Auth Fix

### 现象与原因

- **现象**：新注册用户登录后页面显示 “Session updated. Please try again.”，F12 中大量接口返回 `401`（如 `/api/auth/me`、`/api/categories`、`/api/receipt/list`），仪表盘一直 Loading。  
- **原因（归纳）**：  
  1. 生产环境未配置 `SUPABASE_SERVICE_ROLE_KEY` 时，后端用 anon key 访问 Supabase，RLS 阻止向 `public.users` 插入新用户，`get_or_create_user_id` 插入失败 → 鉴权失败 → 401。  
  2. 生产日志后续发现：`handle_new_user()` 向 `user_class` 写入字符串 `'free'`，而迁移 053 已将 `users.user_class` 改为 `smallint`，导致 `invalid input syntax for type smallint: "free"`。

### 处理与代码

- **agent-tbd**  
  - **NEW_USER_401_SESSION_UPDATED_20260308.md**：诊断文档，说明根因、确认 Service Role Key、临时补救（同步脚本）、以及 062 对 `handle_new_user()` 的修复说明。

- **backend/scripts/sync_firebase_users_to_supabase.py**（新建）  
  - 一次性脚本：从 Firebase Auth 拉取用户列表，按 `firebase_uid` 查找或按邮箱关联，否则插入 `public.users`。  
  - 用法：本地 `python -m backend.scripts.sync_firebase_users_to_supabase`；生产可加 `--production` 或 `--env-file backend/.env.production`。  
  - 可选 `--dry-run` 仅列出 Firebase 用户不写库。  
  - 依赖 Service Role 才能插入/更新 users。

- **数据库迁移 062**  
  - **062_fix_handle_new_user_user_class_smallint.sql**：重写 `handle_new_user()`，插入 `user_class = 0`（smallint）而非 `'free'`，避免类型错误。

### 相关后端

- **jwt_auth.py**：Firebase token 校验后调用 `get_or_create_user_id`；失败时记录日志并返回 401。  
- **firebase_auth.py**：`get_or_create_user_id` 查/插 users 逻辑。  
- **supabase_client.py**：`_get_client()` 使用 service role 或 anon key。

### 前端

- **auth/callback/page.tsx**：与登录回调/重定向相关调整。  
- **dashboard/page.tsx**：收到 401 时展示 “Session updated. Please try again.”（`sessionRefreshedHint`）。  
- **dashboard/developer/page.tsx**：开发者页与鉴权/调试相关小改动。
