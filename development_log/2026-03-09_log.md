## 一、用户分类树重构 | User Category Tree Restructure

### 数据库迁移 064–066

- **064_user_category_tree_restructure.sql**
  - `user_categories`：新增 `sort_order`、`is_locked`（L1 从系统复制为锁定，不可改名/删）。
  - `record_items`：新增 `user_category_id`（用户展示用，可与系统 `category_id` 不同）。
  - 新增 `seed_user_default_categories(p_user_id)`：将系统 L1/L2/L3 写入用户分类树；L1 为 `is_locked=true`。
  - 新增 `resolve_system_to_user_category(p_user_id, p_system_category_id)` 及 `resolve_user_categories_for_receipt(receipt_id)`：系统分类 → 用户分类解析、按小票回填 `user_category_id`。
  - `handle_new_user()` 触发器：新用户注册后自动执行 seed。
  - `record_items_enriched` 视图：增加用户分类字段；RLS 与索引调整。

- **065_backfill_user_categories.sql**
  - 为所有已有用户执行 `seed_user_default_categories`（尚无 user_categories 的用户）。
  - 对已有 `record_items` 回填 `user_category_id`（通过 `resolve_user_categories_for_receipt`）。

- **066_sync_system_categories_to_user.sql**
  - 新增 `sync_system_categories_to_user(p_user_id)`：当管理员新增系统 L1 后，为老用户增量同步缺失的 L1（及 L2/L3）到其用户分类树，避免仅依赖 idempotent seed 导致老用户拿不到新 L1。

### 后端服务与 API

- **backend/app/services/categories/user_categories_service.py**（新建）
  - 用户分类树 CRUD：`get_user_categories`、`get_user_category`、`create_user_category`、`update_user_category`、`delete_user_category`。
  - `seed_user_default_categories_if_needed(user_id)`：首次访问时若无分类则 seed，并可选调用 `sync_system_categories_to_user` 同步新增系统 L1。
  - 与 064 中 resolve/seed 等函数配合，保证 L1 不可创建/改名/删除（is_locked）。

- **main.py**
  - `GET/POST /api/me/categories`：当前用户分类树列表与创建子分类。
  - `PATCH/DELETE /api/me/categories/{cat_id}`：更新名称/排序、删除（仅非锁定节点）。
  - `GET /api/categories`：改为返回当前用户的分类树（内部先 `seed_user_default_categories_if_needed`），供下拉/选择器使用。

- **receipt_categorizer.py**（及 **supabase_client.py**）
  - 分类写入/解析逻辑对接 `user_categories` 与 `record_items.user_category_id`（系统分类 → 用户分类解析、回填）。

## 三、管理员失败小票分类与用户升级 | Admin Failure Kind & User Escalation

### 数据库 068

- **068_admin_failure_kind_and_receipt_escalations.sql**
  - `receipt_status`：新增 `admin_failure_kind`（text，可为 `first_round_fail` | `user_escalated` | `vision_fail` | `escalation_fail` 等），标识为何进入管理员「失败小票」列表。
  - 新建表 `receipt_escalations`：用户点击「升级/Escalate」时记录一条（receipt_id, user_id, notes, created_at），便于管理员查看用户反馈。

### 后端

- **main.py**
  - `POST /api/receipt/{receipt_id}/escalate`：用户提交升级与备注；写入 `receipt_escalations`，并将该小票 `admin_failure_kind` 设为 `user_escalated`，便于管理员筛选「用户升级」类失败。

- **supabase_client.py**
  - 新增 `create_receipt_escalation` 等与 068 表结构对应的写入/查询。

- **failed_receipts_service.py**
  - 列表与详情支持按 `admin_failure_kind` 筛选/展示；拉取并返回 `receipt_escalations` 的备注（如 `escalation_notes`）供管理端展示。

### 前端

- **admin/failed-receipts**
  - 列表与详情页展示 `admin_failure_kind`（如 failure_kind_label）及用户升级备注（escalation_notes）。
- **admin/layout.tsx**
  - 增加管理端导航入口（如 failed-receipts、prompts）。
- **admin/prompts/**（新建）
  - 管理端 Prompt 查看/管理页 `page.tsx`。
