## 一、异步上传架构 | Async Upload Architecture (03-24)

### 问题
原有 `/api/receipt/workflow` 和 `/api/receipt/workflow-vision` 端点为同步阻塞：用户上传图片后需等待完整 OCR + LLM + 分类流程结束才能收到响应（通常 10-30 秒）。

### 方案
将上传拆分为两阶段：

1. **同步预检 `_run_vision_pre_check_sync()`**：图片压缩、去重检查、限速、创建 DB 记录、保存图片 → 立即返回 `receipt_id`
2. **异步后台 `_run_vision_background()`**：`asyncio.create_task()` fire-and-forget，运行完整 vision workflow

**统一入口 `_handle_async_vision_upload()`**：两个端点 (`/workflow`, `/workflow-vision`) 共享同一个异步处理逻辑。

### 影响
- 前端上传后立即收到 `{"status": "processing", "receipt_id": "..."}` → 可轮询 `GET /api/receipt/{id}` 获取结果
- 用户体验从"等 30 秒白屏"变为"即时反馈 + 后台处理"

### 同步清理
- 移除 `/api/receipt/openai-llm` 端点（OpenAI pipeline 已于 03-21 废弃）
- 移除 `process_receipt_with_llm_from_docai` 引用

## 二、Firebase 线程安全 | Firebase Thread Safety (03-25)

### 问题
异步上传引入 `asyncio.to_thread()`，多个后台线程并发初始化 Firebase Admin SDK → 偶发 `already exists` 异常。

### 修复
- `_get_firebase_app()` 加 `threading.Lock()` + double-check locking
- 优先 `firebase_admin.get_app()` 复用已有实例，避免重复 `initialize_app()`

## 六、L1 分类系统重构 | L1 Category System Restructure (03-25, `restructure/category-level`)

### 背景
原有架构：admin 手动选三级分类（L1 → L2 → L3）→ confirm。新架构：admin/系统只负责 L1，用户自建 L2+。

### Migration 070: 系统分类重构
- 将所有 `record_items`、`products`、`product_categorization_rules`、`classification_review` 中指向 L2/L3 的 `category_id` 提升到其 L1 父级
- 停用所有 L2/L3 系统分类（`is_active = FALSE`）
- 重命名 L1：`grocery→groceries`、`household→household supplies`、`health→medical`
- 新增 11 个 L1：snacks & beverages, restaurants, home & furniture, electronics, clothing & apparel, transportation, education & office, entertainment, services & fees, childcare, garden
- 更新 `seed_user_default_categories()` 和 `sync_system_categories_to_user()` 仅处理 L1
- 回填所有现有用户的 `user_categories`

### Migration 071: 分类 Prompt 更新
- 17 个 L1 分类，每个含详细定义、示例、边界说明
- 决策规则：商户覆盖（餐厅全归 Restaurants）、商店上下文、主要用途优先
- LLM 输出格式从 `category_i/ii/iii` 简化为单个 `category` 字段

### 前端：Admin Classification Review
- 三列 Category I/II/III dropdown 简化为单个 Category dropdown（L1 only）
- Confirm 验证从"必须选 Category III"改为"必须选 Category"
- 筛选器同步简化

### 后端：LLM 分类服务
- `classification_llm.py`：`_path_to_category_id()`（L3 路径查找）→ `_name_to_l1_category_id()`（L1 名称查找）
- 兼容新旧 LLM 响应格式（`"category"` 和 `"category_i"`）
