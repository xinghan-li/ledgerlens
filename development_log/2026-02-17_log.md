## 一、Classification Review 后端体系 | Classification Review Backend

### 1.1 表与入队 | Table & Enqueue

- **Migration 025**：新建 `classification_review` 表（raw_product_name, normalized_name, category_id, store_chain_id, source_record_item_id, status 等），索引含 pending 按 (raw_product_name, store_chain_id) 去重。
- **supabase_client**：`enqueue_unmatched_items_to_classification_review(receipt_id)` — 从 record_items 取 `category_id IS NULL` 的行，按 (raw_product_name, store_chain_id) 去重后插入 CR，同品名同店链仅一条 pending。
- **receipt_categorizer**：`categorize_receipt` 在 save_receipt_items 成功后调用上述 enqueue，小票成功分类后未命中项自动进审核队列。

### 1.2 CRUD 与 Confirm API

- **classification_review_service**：列表（带 category_name/path、store_chain_name）、单条、PATCH（normalized_name/category_id/store_chain_id/size_quantity/size_unit/package_type/status）、Confirm。
- **Confirm 逻辑**：校验 normalized_name + category_id；`normalize_name_for_storage` 规范化；可选相似名检查（rapidfuzz），高相似且未传 `force_different_name` 则返回 409 + similar_to；写入 **product_categorization_rules**（upsert，times_matched/last_matched_at）；写入 **products**（按 normalized_name + size_quantity + size_unit + package_type + store_chain_id 唯一，update 或 insert）；**回填 source record_item** 的 product_id、product_name_clean、category_id、unit，并调用 `aggregate_prices_for_date` 刷新 price_snapshots；最后将 CR 行标为 confirmed。
- **main.py**：`GET/PATCH /api/admin/classification-review`，`GET suggest-normalized`，`POST .../confirm`，`DELETE .../{cr_id}`（硬删单行）。

### 1.3 LLM 预填与 Prompt

- **Migration 026**：在 prompt_library 插入 `classification` 系统 prompt，用于根据 raw_product_name 与门店上下文推断 category I/II/III、size、unit_type，输出 JSON schema。
- **prompt_loader**：`load_classification_prompt` 支持 classification 场景。
- **classification_llm**：`suggest_classifications(raw_product_names, store_chain_name)` 调 Gemini，返回每条的 category path、size、unit_type、package_type；`_path_to_category_id` 将 path 解析为 categories 表 level-3 的 id，供前端或批量预填使用。

### 1.4 Size / Unit / Package 与 products 按门店唯一

- **Migration 027**：products、classification_review 将 size/unit_type 拆为 **size_quantity、size_unit、package_type**；products 唯一约束改为 (normalized_name, size_quantity, size_unit, package_type)。
- **Migration 028**：categories 的 name、path 统一为小写。
- **Migration 029**：size_quantity 改为 NUMERIC(12,2)；products 增加 **store_chain_id**，唯一约束改为 (normalized_name, size_quantity, size_unit, package_type, store_chain_id)，NULL store_chain_id 视为同一桶；重建 record_items_enriched 等依赖视图。

## 三、Admin 前端整站 | Admin Frontend

### 3.1 布局与权限

- **admin/layout.tsx**：取 session，请求 `GET /api/admin/classification-review?limit=1` 做 Admin 权限校验；403 时提示「需要 Admin 或 Super Admin 权限」并链回 Dashboard；通过后渲染导航（分类审核、分类管理、门店审核、失败小票、Dashboard）与子页面。

### 3.2 分类审核页 | Classification Review Page

- **classification-review/page.tsx**：状态筛选（pending/confirmed/unable_to_decide/deferred/cancelled）、分页；表格列：raw_product_name、normalized_name（可编辑）、**Category I / II / III 级联下拉**（数据来自 GET categories）、size_qty/unit/package（blur 时 PATCH）、status 下拉；**筛选**：Category I/II/III、unit、package，清除筛选；排序；操作列：pending 时 **Confirm**，confirmed/cancelled/deferred 时 **重开**，每行 **删除**（硬删）。Confirm 若 409 且返回 similar_to，展示「与 XXX 相似」并提供「坚持使用」（force_different_name）与「取消」。

### 3.3 分类管理页 | Categories Page

- **categories/page.tsx**：拉取 GET categories（active_only=false），树状展示（根 + 按 parent_id 分组）；新增分类（选父节点、输入 name、显示 level）、行内编辑 name（PATCH）、软删（DELETE）。

### 3.4 失败小票 | Failed Receipts

- **failed-receipts/page.tsx**：列表（上传时间、状态/阶段、失败原因）、操作列「修正」+「删除」；删除前确认，调用 DELETE /api/admin/failed-receipts/{id}，成功后刷新列表。
- **failed-receipts/[id]/page.tsx**：单条失败小票手动修正；prefill 来自 GET /api/admin/failed-receipts/{id}（summary + items）；展示小票图片（raw_file_url 经 /api/admin/receipt-image 拉取）；表单编辑 store_name、receipt_date、subtotal/tax/total、每行 product_name/quantity/unit_price/line_total 等；提交调用 POST .../submit，成功后跳转或提示。

### 3.5 门店审核 | Store Review

- **store-review/page.tsx**：门店候选列表与审核（依据 store_review_service 与后端 store-review API）。

### 3.6 Dashboard

- **dashboard/page.tsx**：增加「Admin」入口；上传小票结果按 `result.success` 显示绿色成功 / 红色失败与 error 文案，避免误显「处理成功」。

## 四、失败小票与 Workflow 修复 | Failed Receipts & Workflow Fixes

- **异常时保存图片**：workflow 主流程 `except` 分支中若有 db_receipt_id，调用 `_save_image_for_manual_review` 并 `update_receipt_file_url`，失败小票详情页可显示图片。
- **失败小票硬删**：`delete_receipt_hard(receipt_id)` 清 api_calls/store_candidates 引用后删 receipt_status（CASCADE 清理 runs/summaries/items）；`DELETE /api/admin/failed-receipts/{receipt_id}`，仅允许 failed/needs_review 状态。
- **receipt_categorizer**：对 Supabase 返回的 `output_payload` 可能为 JSON 字符串、`.data` 形态不统一做防御，`_ensure_dict` / `_single_row` 规范化后再 `.get()`，修复 `'str' object has no attribute "get"`。
