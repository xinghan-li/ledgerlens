## 一、现象与排障结论 | Symptoms & Root Causes

### 1. 用户分类：`user_categories_user_parent_name_key`（23505）

- **现象**：Error Reporting / Sentry 大量 `duplicate key ... dining`，堆栈指向 `seed_user_default_categories_if_needed` → `sync_system_categories_to_user` RPC。
- **根因**：唯一索引为表达式索引 `(user_id, COALESCE(parent_id, nil_uuid), name)`；`INSERT ... ON CONFLICT DO NOTHING` 未指定冲突目标时，PostgreSQL 无法可靠匹配该索引，并发或「同名已存在」场景下会抛出 23505。
- **修复**（代码库）：`backend/database/073_fix_user_categories_on_conflict.sql` 中为 `seed_user_default_categories` 与 `sync_system_categories_to_user` 使用显式冲突目标：
  `ON CONFLICT (user_id, (COALESCE(parent_id, '00000000-0000-0000-0000-000000000000'::uuid)), name) DO NOTHING`。
- **应用侧降噪**：`user_categories_service.py` 中将 RPC 失败从 `logger.exception` 改为 `logger.warning(..., exc_info=True)`，避免已知幂等竞态被当成 Error 级告警。

**部署注意**：需在 Supabase / 生产数据库执行上述 migration；未执行前旧错误仍可能出现。

### 2. 「无 username 用户上传失败 / 全员上传失败」与分类错误的关系

- **澄清**：分类 seed 异常在 Python 侧已被 `try/except` 吞掉，**单就后端设计而言不应阻断** `POST /api/receipt/workflow-vision` 的预检与建单；Error Reporting 里看到的主要是分类 RPC，容易让人误以为「上传被这一条打死」。
- **仍需单独排查**：
  - **重复账号**：Firebase `get_or_create_user_id` 在 `firebase_uid` / email 对不上时可能新建 `users` 行，小票挂在「新 id」上，老账号列表为空（需在 DB 按 email 查是否多行）。
  - **真正的 pipeline 失败**：vision 后台任务若抛错，多为 **Cloud Run 普通日志**，不一定进 Sentry；应到 **Logs Explorer** 搜 `Gemini`、`vision-async`、`[vision]` 等。

### 3. Gemini 模型与约 3 月下旬起「全员挂掉」的假设

- **本地未提交的 `config.py` diff**：默认模型由 `gemini-1.5-flash` 改为 `gemini-2.5-flash`，并增加 `CONFIDENCE_THRESHOLD`。
- **假设**：线上若仍依赖已废弃或行为变更的旧模型名 / 旧 API 行为，主链路 vision 调用会持续失败；错误多落在 **logger.error** 路径而非未捕获异常。
- **建议动作**：
  1. 生产环境显式设置 `GEMINI_MODEL=gemini-2.5-flash`（或当前 Google 文档推荐的等价 Flash 模型名）。
  2. 合并并部署本地对 `config.py`、`gemini_client.py`（metadata schema 含 `confidence`）、`workflow_processor_vision.py`（confidence gate）的修改，与 DB migration `072`/`073` 的版本预期保持一致。

## 二、运维观测 | Operations

### Supabase PostgREST 「unhealthy」短尖峰

- **现象**：Logs 中短期内多次 `Received a schema cache reload message on the 'pgrst' channel`、`Connection pool initialized`、`Schema cache loaded`。
- **解释**：对数据库执行 `CREATE OR REPLACE FUNCTION` 等 DDL 会触发 PostgREST 重载 schema；**通常为数秒内的正常抖动**，非长期故障。
